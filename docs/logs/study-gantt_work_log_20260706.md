## 修正1: v6が反映されない／スパンで内容が食い違う

### 根本原因
直近コミット `e633b1b` で `public/schedule.md` を `public/1Q_schedule_v6.md` に**リネーム**していた。
アプリは `public/schedule.md` を fetch するため 404 になり、localStorage / SW の**古いキャッシュ**へ
無言でフォールバック。加えて `sw.js` の `CACHE` が `v1` から一度も上がらず `app.js` 等が
cache-first で恒久固定され、コード更新がPWAに届かない構造だった。

### 対応
- `public/1Q_schedule_v6.md` → `public/schedule.md` に `git mv`（fetchパスを固定に統一）。
- **`app.js` `init()`**: `fetch('public/schedule.md?_='+Date.now(), {cache:'no-store'})` で
  キャッシュバスティング＋network-first。取得チェーンを `fetch → localStorage → 内蔵DEFAULT_MD` に整理。
- **`sw.js`**: `CACHE` を `study-gantt-v2` に更新（activate時に旧キャッシュ破棄）。
  `schedule.md` に加え `app.js`/`style.css`/`index.html`/`/` を network-first 化（`isNetworkFirst()`）。
- 週/月/全体は元々同一の `state.schedule`（単一パース結果）から描画。食い違いは古いキャッシュが原因で、
  上記で解消（ビュー側コードの変更は不要）。

## 修正4: ナビゲーション可動域が7月で頭打ち（追記への対応）

### 調査結果
- `◀▶` ハンドラは **navDate を何とも比較していなかった**。クランプ・保存値・today/meta比較・化石定数はコード上に一切なし。
- `app.js` の**全リビジョンを grep** しても navDate の min/max・`07-11`/`07-26` 等の日付リテラル・
  clamp/limit/bound は歴史上一度も存在せず → **仮説A（化石定数）・仮説B（todayクランプ）はコードレベルで否定**。
  localStorage にも範囲系キーなし（AK確認済み）。
- 実 `render()` で v6 を描画すると月Aug=25本・週8/16=6本など**7月以降のバーは正常描画**（Sep=0本＝8/29終了で正しい）。
  → 観察された「7月頭打ち」は**古いスケジュールのキャッシュ表示**が原因で、ナビロジックのバグではない（修正1のSW対応で解消）。

### 対応（期待動作の実装）
- `getNavBounds()` を追加：現在の `viewMode` と `meta.startDate〜endDate` から、週ビュー=週頭・月ビュー=月頭の
  可動域 min/max を**毎回算出**。
- `render()`：`◀▶` を `getNavBounds()` 基準で `disabled` 制御（全体ビューは従来どおり両方無効）。
- navハンドラ：新 navDate を meta 範囲へクランプ（空の月へ流出しない）。
- 化石定数は存在しないため撤去対象なし。可動域は常に読込中スケジュールに追随（v6なら▶は8月で止まり、
  古い版なら7月で止まる＝版ズレが操作でも即座に分かる）。

## 注意点・申し送り

- **SWのワンリロード遅延**：新SW(v2)はデプロイ後の初回アクセスで install→activate され、
  実際に新コードが効くのは**次のリロードから**（Service Worker仕様の既知挙動）。iPhone PWAは一度リロードすれば以降は最新が届く。
- **`DEFAULT_MD` と `public/schedule.md` の同期**：今後スケジュール更新時は両方を更新する（内蔵は最終フォールバック）。
- **版の識別はファイル名でなく md メタ情報の終了日で行う運用**に変更。ファイル名 `public/schedule.md` は固定し、リネームしない。
- 反映確認の勘所：フッター `版: 1Q 〜2026-08-29` ＋ 月表示で▶が8月まで動くこと。7月で止まる場合はキャッシュ未切替の合図。
