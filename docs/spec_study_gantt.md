# 学習管理アプリ（Study Gantt） — 仕様書

## 概要

| 項目 | 内容 |
|------|------|
| **アプリ名** | Study Gantt |
| **URL** | `sinacost-byte.github.io/study-gantt/` |
| **目的** | 「Self-Study with AI University」の学習スケジュールを俯瞰し、進捗を管理する |
| **ユーザー** | AK（1人。個人学習用） |
| **端末** | iPhone / iPad（PWA）メイン、PC（MacBook Air / Chrome）でも閲覧可 |
| **設計思想** | ガントチャート一本でシンプルに。「見る」と「完了をつける」だけ。書き込まない、いじらない |

---

## 背景

- AKは普段から仕事でガントチャート（Brabio）を使っており、スケジュールの俯瞰・進捗把握にガントチャートを好む
- 本プロジェクトは3Quarter制（1Qは2026-08-29まで延長済み）で、1Qに3科目（A/B/C）を並行して進める
- シラバスは Markdown ファイルで管理されており、Claude と相談しながら更新する運用
- スケジュールの「原本」は Markdown。アプリはそれを表示・可視化する読み取り専用のビューワー

---

## コンセプト：Single Source of Truth

```
Markdown（シラバス） ──読み込み──→ アプリ（表示＋進捗管理）
       ↑                              ↓
  Claude と相談して修正          進捗データはアプリ側に保存
```

- **スケジュール・タスクの定義**: Markdown が唯一の原本。アプリ側では編集しない
- **進捗データ（完了/未完了）**: アプリ側（localStorage）に保存する（mdには書き戻さない）
- Markdown を修正して再読み込みすると、スケジュールが更新される。進捗データは保持される

## 機能

### コア機能

1. **ガントチャート表示**: 科目×週のスケジュールをガントチャートで表示する
2. **表示スパン切り替え**: 週 / 月 / 全体 の3段階で表示範囲を切り替えられる
3. **期間ナビゲーション**: ◀ ▶ ボタンで週/月単位でページ送り。「今日」ボタンで現在週/月に戻る
   - 可動域はスケジュールmdのメタ情報（開始日〜終了日）から毎回算出（`getNavBounds()`）。端で◀▶が disabled（opacity 0.3）になり、空の期間へは出られない。全体ビューでは◀▶とも無効
   - 週・月ビューは1週間／1ヶ月ぶんだけを描画する**ページング方式**。チャートのスワイプは描画範囲内のみで、期間の移動は◀▶で行う（意図した設計）
4. **進捗管理（タスク単位）**: 各タスクのチェックボックスをタップして「完了」をつける
5. **進捗管理（週単位・自動）**: その週のタスクがすべて完了すると、週のバーが自動で「完了」表示になる（「完了」バッジ付き）
6. **進捗管理（科目単位・自動）**: 科目内の全タスクが完了すると、科目バーにも「完了」バッジが付く
7. **マイルストーン表示**: 大きなゴールと到達点を◆マーカーで表示する
8. **「今日」の表示**: ガントチャート上に赤い縦線で「今日」の位置を表示する
9. **折りたたみ**: 左列の科目⇒週⇒タスクを3段階で折りたたみ/展開できる
10. **Markdown 読み込み**: 📁ボタンから新しいスケジュールmdファイルを読み込んで更新できる
11. **ラベルツールチップ**: 見切れたラベルの全文を確認できる（下記参照）
12. **版表示（フッター）**: スケジュールの版・取得元・取得日時・コード版を常時表示する（下記参照）

### スクロール位置ポリシー（2026-07-16導入）

`render()` は `app.innerHTML` の全書き換え方式のため、素のままでは操作のたびにスクロール位置が先頭へ戻る。これを `render(opts)` のモードで制御する。**縦（scrollTop）はどのモードでも保持**。横（scrollLeft）だけ操作の性質で分ける。

| モード | 縦 | 横 | 適用される操作 |
|--------|----|----|----------------|
| `preserve`（既定） | 保持 | 保持 | ✅トグル、折りたたみ/展開 |
| `reset-x` | 保持 | 左端 | ◀▶（期間送り）、ビュー切替 |
| `center-today` | 保持 | 今日線をセンタリング | 「今日」ボタン、起動時、md手動読込 |

- 今日線のセンタリングは `scrollTodayIntoView()` ヘルパー（横方向のみ担当）
- 内容が縮んだ場合の縦のはみ出しはブラウザのクランプに任せる

### ラベルツールチップ

左列のラベルは幅の制約で見切れることがある。全文を確認する操作：

| 行タイプ | 操作 | 動作 |
|---------|------|------|
| task | ラベル部分をタップ | 全文ツールチップ表示 |
| week | ラベル部分を長押し（500ms） | 全文ツールチップ表示 |
| section | — | そのまま表示（短文のため見切れない） |

- ツールチップ外タップで閉じる
- 長押し検知後は折りたたみを発火しない（フラグ制御）
- iOS長押し時のコンテキストメニューは `-webkit-touch-callout: none` で抑止

### 版表示（フッター2段目 `.footer-meta`）

`版: 1Q 〜2026-08-29 / 取得元: サーバー取得 · 7/16 17:23 · app 20260716-1`

- **版**: 表示中スケジュールの Quarter と終了日（版の識別は**ファイル名でなくmdメタ情報の終了日**で行う）
- **取得元**: `サーバー取得` / `キャッシュ(前回取得)` / `内蔵フォールバック` / `手動読込`。内蔵フォールバック時は黄背景＋⚠で明示
- **app版スタンプ**: `APP_VERSION` 定数。動いているコードの版を可視化し、Pages配信遅れ・SWワンリロード遅延を検知する

### つけない機能（意図的に省略）

- スケジュール編集（アプリ側ではしない。md を修正して再読み込み）
- 詳細なタスク記入・メモ・ファイル添付
- タスクの詳細画面
- カレンダー表示（ガントチャートの表示スパン切り替えで代替）
- 通知・リマインダー（Apple 純正リマインダーの役割）
- 他ユーザーとの共有・コラボレーション
- Google カレンダー連携
- 週・月ビューの連続スクロール（ページング方式を維持）

## 画面構成

### メイン画面（ガントチャート）

```
┌──────────────────────────────────────────────────┐
│  [週] [月] [全体]   ◀ 7/12 − 7/18 ▶   [📅今日] [📁md読込] │
├──────────────────────────────────────────────────┤
│  1Q              │7/12(日) 7/13(月) 7/14(火)  … │
│  ─────────────────┼──────────────────────────────│
│  ▶ オリエンテーション │                              │
│  ▼ A: 技術革命史  │ ████████████████████████████ │
│    ▶ Week 1: 技術… │                              │
│    …               │         ↓今日                │
├──────────────────────────────────────────────────┤
│  Self-Study with AI University        1/146 完了  │
│  版: 1Q 〜2026-08-29   取得元: サーバー取得 · … · app … │
└──────────────────────────────────────────────────┘
```

- 左列：科目名 → 週タイトル → タスク（3段階折りたたみ）
  - ▼/▶ で展開・折りたたみをトグル
  - 初期状態：科目は展開、週は折りたたみ（オリエンテーションのタスクは表示）
- 右側：時間軸にバーを表示
- 表示スパンに応じてバーの粒度が変わる：
  - **週表示**: 日単位（曜日付き）
  - **月表示**: 週単位
  - **全体表示**: 月単位。1Q全体が俯瞰できる
- フッター1段目：アプリ名と完了タスク数。**完了カウントは折りたたみ状態に依存せず、パース結果の全タスクを集計**（`collectAllTaskIds()`）
- フッター2段目：版表示（前述）

### タスク完了の操作

- チェックボックスをタップすると「完了/未完了」がトグルする
- バー上をタップしても完了をトグルできる
- 完了 = チェックボックスが塗りつぶされ、バーの色が淡色に、チェックマーク付き。ラベルに取り消し線
- 誤タップした場合はもう一度タップで戻せる
- **トグルしても画面のスクロール位置は動かない**（スクロール位置ポリシー `preserve`）

## データ構成

### スケジュールの取得チェーン（network-first）

起動時、以下の順で取得する。どれを表示しているかは `scheduleSource` としてフッターに常時表示：

1. **サーバー取得**: `fetch('public/schedule.md?_='+Date.now(), {cache:'no-store'})` — キャッシュバスティング付き。成功したら localStorage にも保存
2. **キャッシュ**: 前回サーバー取得に成功した localStorage のコピー（オフライン時）
3. **内蔵フォールバック**: ソースコード埋め込みの `DEFAULT_MD` 定数（fetch・キャッシュ双方失敗時のみ。⚠表示）

**運用ルール**:
- スケジュール更新は `public/schedule.md` の**内容差し替え**で行う。**ファイル名の変更は禁止**（fetchパスが固定のため。過去にリネームで404→無言フォールバックの障害）
- スケジュール更新時は `DEFAULT_MD` も同内容に更新する（両者の同期を保つ）

### 入力：Markdown フォーマット

```markdown
# 1Q スケジュール

## メタ情報
- 開始日: 2026-03-28
- 終了日: 2026-08-29
- Quarter: 1Q
- 週の開始曜日: 日曜日

---

## オリエンテーション（2026-03-28 〜 2026-04-04）
- [ ] カリキュラム全体の確認・学習環境セットアップ

## A: 技術革命史

### Week 1: 技術革命の全体地図（2026-04-05 〜 2026-04-11）
- [ ] 講義: 技術革命のリスト
- [x] 完了済みで初期化したいタスク

## マイルストーン
- 2026-04-04: オリエンテーション完了
```

**パースルール**:
- `## セクション名` → 科目グループ（オリエンテーション、A、B、C、マイルストーン）。それ以外の `##`（週間リズム等）は無視される
- `### Week N: タイトル（YYYY-MM-DD 〜 YYYY-MM-DD）` → 週。括弧内の日付でバーを配置
- `## セクション名（YYYY-MM-DD 〜 YYYY-MM-DD）` → 日付付きセクション（オリエンテーション等）
- `- [ ] テキスト` → 未完了タスク／`- [x] テキスト` → 完了済みタスク（md上での初期状態指定用）
- `## マイルストーン` 配下の `- YYYY-MM-DD: ラベル` → マイルストーン
- `## メタ情報` 配下 → Quarter全体の設定情報。**◀▶の可動域はここから算出**
- 科目の開始日・終了日は所属する週の先頭・末尾から自動算出

**タスクIDの生成ルール**:
- `{Quarter}-{科目}-{WeekN}-task{N}` の形式で自動生成（週IDは `Week1` 形式）
- 例：`1Q-A-Week1-task1`、`1Q-B-Week3-task2`、`1Q-ORI-task1`（オリエンテーション）
- **位置ベース**：md を再読み込みしたとき、同じ位置のタスクは同じIDが振られ、進捗データが保持される。週内のタスクの並び替え・挿入はID がずれるため注意

### 出力（進捗データの保存）

- 進捗データは **localStorage** に保存
- ストレージキー: `sg-completed`（進捗）、`sg-schedule-md`（前回取得したスケジュールのキャッシュ）
- データ形式:
  ```json
  { "1Q-ORI-task1": true, "1Q-A-Week1-task1": true }
  ```
- Markdown を再読み込みしても、タスクIDが一致する限り進捗データは保持される
- **ブラウザのサイトデータ削除時には進捗が失われる**（既知の制約。端末ごとに独立保存であり、端末間同期はない）

## 日付処理の注意事項

- 日付文字列のパース: `new Date(s + 'T00:00:00')` で時刻をローカル深夜0時に正規化
- `getWeekStart()`: 週の開始日（日曜）を取得する際に `setHours(0,0,0,0)` で時刻を正規化する（`navDate` が現在時刻を持つため、正規化しないと `dayDiff` のround で1日ズレる）
- `dayDiff()`: `Math.round((b - a) / 86400000)` で日数差を計算

## 技術要件

| 項目 | 内容 |
|------|------|
| **形式** | PWA（Progressive Web App） |
| **実装** | バニラ HTML / CSS / JavaScript（`index.html`, `style.css`, `app.js`, `sw.js`） |
| **対応端末** | iPhone / iPad（Safari / PWA）メイン。PC は Chrome（MacBook Air） |
| **ホスティング** | GitHub Pages（public リポジトリ: `sinacost-byte/study-gantt`） |
| **データ保存** | localStorage（`sg-completed` / `sg-schedule-md`） |
| **初期データ** | スケジュールmdをソースコードに埋め込み（`DEFAULT_MD`定数） |
| **オフライン** | Service Worker によるキャッシュで対応 |

### Service Worker（キャッシュ戦略）

- **コア資産は network-first**（`schedule.md` / `app.js` / `style.css` / `index.html` / `/`）：オンライン時は常に最新を取得し、成功したらキャッシュ更新。オフライン時のみキャッシュへフォールバック
- **その他（アイコン等）は cache-first**
- `CACHE` 定数によるバージョニング。activate 時に旧キャッシュを破棄

### 運用ルール（コード変更時）

- app.js を変更するコミットでは `APP_VERSION` を必ず更新する
- デプロイ（push）ごとに sw.js の `CACHE` をバンプする
- **SWのワンリロード遅延**: 新SWはデプロイ後の初回アクセスで install→activate され、実際に新コードが効くのは次のリロードから。受け入れ確認は2回リロード後、フッターの app 版スタンプで新コード到達を確認してから行う
- Pages配信遅れの一次切り分け: ブラウザで `…/study-gantt/app.js` を直接開き、期待する関数名や `APP_VERSION` を検索する
